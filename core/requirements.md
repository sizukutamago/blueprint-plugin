# Requirements Workflow

ユーザーへの構造化インタビューを通じて要件定義を行うワークフロー。
ユーザーがビジネス判断、AI がヒアリング・構造化・品質検証を担当する。

> **前提**: `blueprint-structure.md` と `id-system.md` を参照。

## ワークフロー（6 ステップ）

### Step 1: コンテキスト読み込み + モード判定

プロジェクトの状態を確認し、greenfield/brownfield を判定する。

```
チェック項目:
- docs/requirements/ の存在（既存要件の有無）
- .blueprint/ の存在（既に /spec 済みか）
- src/ の存在 + ファイル数（brownfield 判定）
- README.md, PRD 等のプロジェクト説明ドキュメント
```

**モード判定基準**:

| 条件 | モード |
|------|--------|
| `src/` なし、または `src/` 内ファイル数 < 5 | greenfield |
| `src/` あり、かつファイル数 >= 5 | brownfield |

**greenfield の場合**:
- Explorer は完全スキップ
- README.md / PRD があれば読み込んでコンテキストとする
- なければ Step 2 に直行

**brownfield の場合**:
- 分析エージェントを 3 つ並列で起動して既存コードベースを調査する

| 分析エージェント | 分析対象 | 出力 |
|----------------|---------|------|
| Tech Stack Analyzer | package.json, tsconfig.json, 設定ファイル | 技術スタック、フレームワーク、ライブラリ一覧 |
| Domain Analyzer | src/ 配下のモデル、型定義、ルーティング | 主要エンティティ、ドメインモデル、画面構成 |
| Integration Analyzer | 外部 API 呼び出し、DB 接続、環境変数 | 外部連携先、NFR の推定 |

3 エージェントの結果をマージして `docs/requirements/.work/context_summary.md` に書き出す。

**docs/requirements/ が既に存在する場合**:
- 既存の user-stories.md を読み込んでユーザーに提示
- 「上書き更新」か「追記」かを確認

### Step 2: インタビュー（Double Diamond + EARS 記法）

ユーザーから要件を引き出すコアフェーズ。
**`/requirements` の最大の価値がここにある。**

#### Double Diamond フロー

```
Diamond 1 — 発散（Discover）
  Q1: プロジェクト概要
  Q2: ターゲットユーザー
  Q3: 主要ユースケース
  Q4: プラットフォーム
      ↓
Diamond 1 — 収束（Define）
  Q5: MVP 優先度（MoSCoW 分類）
  Q6: 非目標（スコープ外の明示）
      ↓
Diamond 2 — 深掘り（Develop）
  Q7: ビジネスルール・制約
  Q8: 成功指標
      ↓
Diamond 2 — 確認（Deliver）
  Q9: 外部連携・データソース（条件付き）
  Q10: NFR・リスク（条件付き）
```

#### 質問ルール

- 全質問は構造化された形式で実行する（自由テキスト質問禁止）
- 1 質問ずつ実行する（まとめて聞かない）
- ユーザーの回答に応じてフォローアップ質問を追加してよい（最大 10 質問の枠内）
- Q9, Q10 は Q7/Q8 の回答内容に応じて条件付き実行

#### EARS 記法の適用

各回答を以下の記法で即座に分類する:

| 記法 | パターン | 用途 | 例 |
|------|---------|------|-----|
| SHALL | `<主語> SHALL <動作>` | 必須機能 | システム SHALL 注文履歴を表示する |
| WHEN-THEN | `WHEN <イベント> THEN <主語> SHALL <動作>` | イベント駆動 | WHEN 在庫が 0 THEN システム SHALL 通知する |
| IF-THEN | `IF <条件> THEN <主語> SHALL <動作>` | 条件分岐 | IF 管理者 THEN システム SHALL 全データを表示する |
| WHERE | `WHERE <制約>` | 非機能制約 | WHERE レスポンスは 500ms 以内 |
| MAY | `<主語> MAY <動作>` | オプション機能 | ユーザー MAY ダークモードを選択する |
| MUST NOT | `<主語> MUST NOT <動作>` | 禁止要件 | システム MUST NOT パスワードを平文保存する |

#### 信頼度レベル

全記録に信頼度を付与する:

| レベル | 意味 | 基準 |
|--------|------|------|
| Blue | ユーザーが明示的に述べた | インタビューで直接回答された内容 |
| Yellow | 妥当な推論 | 回答から合理的に導出できる内容 |
| Red | AI の推測 | ユーザーが言及していないが補完した内容 |

#### 終了条件

- 最大 **10 質問**（フォローアップ含む）
- ユーザーが「十分」「次に進んで」等で終了宣言
- 未解決の論点は `open_questions` リストに退避して次へ進む

**出力**: インタビュー記録（`docs/requirements/.work/interview_log.md`）

### Step 3: 構造化（Epic → Feature → Story 階層化）

インタビュー結果 + brownfield コンテキスト（あれば）から、ユーザーストーリーの階層構造を生成する。

```
処理手順:
1. ペルソナの確定
   - インタビューで明示されたペルソナ → Blue
   - 推定ペルソナ → Yellow
   - ID: P-XXX

2. Epic の特定
   - ビジネスゴール / 大機能領域ごと → Epic-XXX

3. Story の特定
   - 各 Epic 内のユーザー操作単位 → US-XXX
   - 1 Story = 1 ユーザー操作（分割の粒度）

4. MoSCoW 分類
   - Must: MVP に必須
   - Should: MVP に望ましい
   - Could: あれば良い
   - Won't: 今回は対象外（Non-Goals から導出）

5. 依存関係の整理
   - Story 間の blockedBy / enables 関係
```

**ストーリーマップの提示フォーマット**:

```
## ストーリーマップ

### ペルソナ
| ID | ペルソナ | 説明 | 信頼度 |
|----|---------|------|--------|
| P-001 | {名前} | {属性・ゴール} | Blue |

### Epic → Story マッピング
| Epic | Story ID | ストーリー | 優先度 | 信頼度 |
|------|----------|----------|--------|--------|
| Epic-001: {名前} | US-001 | {タイトル} | Must | Blue |
| Epic-001: {名前} | US-002 | {タイトル} | Must | Yellow |
| Epic-002: {名前} | US-003 | {タイトル} | Should | Blue |

### Non-Goals
- {スコープ外の項目}
```

ユーザーの承認を得てから次へ進む（修正がある場合は反映して再提示）。

**出力**: ストーリーマップ（`docs/requirements/.work/story_map.md`）

### Step 4: ユーザーストーリー生成

承認されたストーリーマップに基づき、正式なユーザーストーリードキュメントを生成する。

**各 Story の生成ルール**:

1. **As a / I want / So that 形式**（必須）
2. **EARS 記法**（必須）: 各 Story に対応する EARS 分類を付記
3. **受け入れ基準（AC）**:
   - Gherkin 形式（Given/When/Then）
   - 正常系 AC: 最低 1 件（必須）
   - 異常系 AC: 最低 1 件（必須）
   - 各 AC に信頼度レベルを付与
   - ID: AC-{StoryID の数字部分}-{連番}（例: AC-001-1）
4. **NFR 紐付け**:
   - WHERE 記法で該当 Story に紐付け
   - NFR-{CAT}-XXX ID は既存 id-system.md の形式に合わせる

**出力先**: `docs/requirements/user-stories.md`（メイン成果物）

出力フォーマットの詳細は `user_stories_format.md` を参照。

### Step 5: 品質チェック + 自動補正

生成したユーザーストーリーの品質を自動検証し、修正する。

#### チェック項目

| チェック | 内容 | 違反時の対応 |
|---------|------|-------------|
| 曖昧語チェック | 禁止語リスト（`quality_rules.md` 参照）が残っていないか | 自動で具体化を試みる |
| EARS 準拠 | 全要件が SHALL/WHEN-THEN/IF-THEN/WHERE/MAY/MUST NOT のいずれかに分類されているか | 自動分類を試みる |
| テスト可能性 | 全 AC に Given/When/Then があるか。定量基準があるか | 定量化を試みる |
| 完全性 | 全 Epic に最低 1 Story、全 Story に正常系+異常系 AC | 不足分を自動補完 |
| 信頼度バランス | Red が全体の 30% を超えていないか | 追加質問を提案（強制しない） |

#### 自動補正ループ

```
品質チェック実行
  ↓
問題あり → 自動修正（1 回目）
  ↓
再チェック → まだ問題あり → 自動修正（2 回目）
  ↓
再チェック → まだ問題あり → ユーザーに報告して確認
  ↓
問題なし or ユーザー承認 → Step 6 へ
```

最大 2 回の自動修正。それでも解決しない項目はユーザーに確認する。

### Step 6: サマリー出力

生成結果をまとめて次のアクションを提示する。

```
## 要件定義完了

### 生成ファイル
- docs/requirements/user-stories.md（メイン成果物）
- docs/requirements/.work/（中間成果物）

### 統計
| 項目 | 件数 |
|------|------|
| ペルソナ | N |
| Epic | N |
| ユーザーストーリー | N |
| 受け入れ基準 | N |
| NFR | N |

### 信頼度サマリー
| レベル | 件数 | 割合 |
|--------|------|------|
| Blue | N | N% |
| Yellow | N | N% |
| Red | N | N% |

### 未解決事項
- {open_questions}

### 次のステップ
Contract を生成するには: `/spec`
- user-stories.md を入力として Contract YAML を生成
```

## /spec との連携

`/requirements` の出力は `/spec` の入力として消費される。

| 連携ポイント | 内容 |
|-------------|------|
| `/spec` Step 2 で自動読み込み | `docs/requirements/user-stories.md` が存在する場合、スコープ確認のベースにする |
| Epic/Story → Contract 候補 | ストーリーマップの構造が Contract 一覧の初期候補になる |
| US-XXX → FR-XXX 導出 | ユーザーストーリー ID から機能要件 ID を導出する |
| 重複質問の回避 | `/requirements` で回答済みの内容は `/spec` のブレストで再質問しない |

## 原則

| 原則 | 説明 |
|------|------|
| ユーザーがビジネス判断 | AI は質問・構造化・品質チェックを担当。要件の優先度・スコープはユーザーが決める |
| 信頼度の透明性 | AI の推測は Red で明示。ユーザーが確認すれば Blue/Yellow に昇格 |
| テスト可能性 | 全要件・AC は検証可能な基準を含む。曖昧な記述は具体化を求める |
| EARS 記法で構造化 | 自然言語の要件を 6 分類で構造化し、テスト導出しやすくする |
| YAGNI | 「将来必要になるかも」より「今回の MVP に必要か」を優先 |

## エラーハンドリング

| エラー | 対応 |
|--------|------|
| インタビューが収束しない | 10 質問上限 + open_questions への退避 |
| brownfield 分析が大規模すぎる | ファイル数 > 500 の場合、ディレクトリ単位のサンプリングに切り替え |
| ユーザーが曖昧な回答を続ける | 具体例を提示して選択させる（「例えば X ですか、Y ですか？」） |
| 既存 requirements との競合 | 既存ファイルを提示し、上書き / 追記 / 中止 を確認 |
| 信頼度 Red が 50% 超 | インタビュー不足と判断し、追加質問を強く推奨 |
